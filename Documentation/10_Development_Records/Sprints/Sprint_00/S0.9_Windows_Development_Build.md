# S0.9 — Windows Development Build

**Proyecto:** Cartridge & Cloud  
**Sprint:** 0  
**Bloque:** S0.9  
**Estado:** In Progress  
**Editor:** Unity `6000.3.18f1`  
**Commit de partida:** `04264a362e33c6b823a51f82434b38d879c3dae1`  
**ADR:** ADR-0007

## 1. Objetivo

Crear, ejecutar y registrar el primer build Windows x64 fuera del Editor sin
introducir gameplay.

## 2. Preparación

1. Confirmar GitHub Desktop limpio.
2. Abrir Unity `6000.3.18f1`.
3. Ejecutar EditMode y PlayMode una vez.
4. Exigir 9/9 PASS.
5. Confirmar consola sin errores rojos.
6. Cerrar Test Runner.

## 3. Crear la carpeta del perfil

En la ventana Project:

```text
Assets/_Project/Settings/BuildProfiles
```

Crear las carpetas que falten desde Unity.

## 4. Crear el Build Profile

1. Abrir `File > Build Profiles`.
2. En `Platforms`, seleccionar Windows.
3. Confirmar que Windows Build Support está instalado.
4. Usar `Copy to new profile`.
5. Guardar el perfil dentro de:
   `Assets/_Project/Settings/BuildProfiles`.
6. Nombrarlo `Windows_Development`.
7. Activar el perfil.

Unity guarda los Build Profiles como assets versionables.

## 5. Scene List del perfil

Activar la lista de escenas propia del perfil y establecer:

```text
0 — Assets/_Project/Scenes/Bootstrap.unity
1 — Assets/_Project/Scenes/MainMenu.unity
2 — Assets/_Project/Scenes/Store.unity
3 — Assets/_Project/Scenes/TestLab.unity
```

Las cuatro escenas deben estar habilitadas.

El perfil de desarrollo incluye `TestLab`. Un futuro perfil de distribución la
excluirá.

## 6. Windows Build Settings

Configurar:

| Ajuste | Valor |
|---|---|
| Architecture | Intel 64-bit |
| Build and Run on | Local Machine |
| Development Build | Activado |
| Autoconnect Profiler | Desactivado |
| Deep Profiling | Desactivado |
| Script Debugging | Desactivado |
| Wait for Managed Debugger | Desactivado |
| Compression Method | LZ4 |
| Copy PDB files | Desactivado |
| Create Visual Studio Solution | Desactivado |

## 7. Player Settings

En el perfil o en `Edit > Project Settings > Player`, confirmar:

| Ajuste | Valor |
|---|---|
| Company Name | `VRM Games` |
| Product Name | `Cartridge & Cloud` |
| Version | `0.0.1` |
| Scripting Backend | Mono |
| Use Player Log | Activado |
| Color Space | Linear |

No cambiar en este bloque:

- Application Identifier.
- Iconos.
- Splash Screen.
- Resolución.
- Input Handling.
- Managed Stripping Level.
- API Compatibility Level.

## 8. Construir

Ruta de salida:

```text
Builds/Windows/Development/CAC_v0.0.1_Sprint0_Dev_Windows_x64_2026-06-22_Build001/CartridgeAndCloud.exe
```

Usar `Build`, no `Build And Run`, para separar creación y validación.

Resultado esperado mínimo:

```text
CartridgeAndCloud.exe
CartridgeAndCloud_Data/
UnityPlayer.dll
UnityCrashHandler64.exe
```

Un Development Build puede incluir archivos adicionales como
`WinPixEventRuntime.dll`.

## 9. Ejecutar fuera del Editor

1. Cerrar o minimizar Unity.
2. Ejecutar `CartridgeAndCloud.exe` desde el Explorador.
3. Mantenerlo abierto al menos 10 segundos.
4. Confirmar que el proceso no se cierra ni se bloquea.
5. Cerrar con `Alt+F4`.

La ventana puede ser negra o vacía. Esto es esperado porque `Bootstrap` no
contiene cámara.

## 10. Player.log

Abrir:

```text
%USERPROFILE%\AppData\LocalLow\VRM Games\Cartridge & Cloud\Player.log
```

Comprobar:

- El archivo existe y tiene fecha de la ejecución.
- Muestra Unity `6000.3.18f1`.
- No contiene crash.
- No contiene excepciones no controladas.
- No contiene fallos de carga de escenas o assemblies.

No se exige que el archivo esté completamente libre de warnings informativos.

## 11. Archivar localmente

Comprimir la carpeta del build como:

```text
Builds/Windows/Development/CAC_v0.0.1_Sprint0_Dev_Windows_x64_2026-06-22_Build001.zip
```

Calcular SHA-256 desde Command Prompt:

```text
certutil -hashfile "CAC_v0.0.1_Sprint0_Dev_Windows_x64_2026-06-22_Build001.zip" SHA256
```

Esta operación no usa Git ni PowerShell.

Registrar hash, tamaño, duración y resultado en:

```text
Documentation/10_Development_Records/Builds/
S0.9_Windows_Development_Build_Record.md
```

## 13. Revisión Git

`Builds/` no debe aparecer en GitHub Desktop porque está ignorada.

Deben aparecer:

- Asset y `.meta` del Build Profile.
- Posibles cambios de ProjectSettings causados por la configuración.
- Documentación v0.3.5.
- Evidencias S0.9.

No deben aparecer:

- Ejecutable.
- Carpeta `_Data`.
- DLL del Player.
- ZIP del build.
- Library, Logs o UserSettings.
