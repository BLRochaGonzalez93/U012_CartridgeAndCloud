# S0.6 — Assembly Definitions y límites de dependencia

**Proyecto:** Cartridge & Cloud  
**Sprint:** 0  
**Bloque:** S0.6  
**Versión:** v001

## Objetivo

Separar el código futuro en ensamblados con dependencias explícitas, impedir
dependencias circulares y mantener las reglas de simulación aisladas de Unity y
de la interfaz.

## Ensamblados

| Assembly | Referencias permitidas | UnityEngine | Propósito |
|---|---|---:|---|
| Domain | Ninguna | No | Reglas, entidades, value objects e invariantes |
| Application | Domain | No | Casos de uso, comandos, consultas, contratos y resultados |
| Infrastructure | Domain, Application | Sí | Persistencia, archivos, adaptadores y servicios técnicos |
| Presentation | Domain, Application | Sí | MonoBehaviours, UI, entrada, cámara y representación |
| Tests.EditMode | Los cuatro anteriores y Test Runner | Sí | Pruebas rápidas y pruebas Editor |
| Tests.PlayMode | Los cuatro anteriores y Test Runner | Sí | Pruebas con ciclo de ejecución |

## Grafo aprobado

```text
Domain
  ↑
Application
  ↑            ↑
Infrastructure Presentation
```

`Infrastructure` y `Presentation` no se referencian entre sí.

La composición de implementaciones concretas se resolverá más adelante mediante
un punto de composición específico; no se introduce una referencia
`Presentation -> Infrastructure` en Sprint 0.

## Reglas obligatorias

1. `Domain` no puede usar `UnityEngine`, `UnityEditor`, UI, escenas ni paquetes.
2. `Application` no puede usar `UnityEngine`, `UnityEditor`, UI ni escenas.
3. `Infrastructure` no puede referenciar `Presentation`.
4. `Presentation` no puede acceder directamente a implementaciones de
   `Infrastructure`.
5. Ningún assembly de producción puede referenciar assemblies de tests.
6. Los scripts nuevos deben estar dentro de una carpeta gobernada por un
   `.asmdef`.
7. No se crearán scripts de proyecto directamente bajo `Assets` ni en una
   carpeta sin assembly.
8. No se añadirán referencias preventivas a Input System, Localization,
   AI Navigation o UI hasta que un script real las necesite.
9. Los nombres de assemblies quedan congelados.
10. Cualquier nueva referencia requiere justificación y revisión del grafo.

## Decisión sobre Shared

Eliminar `Assets/_Project/Scripts/Shared`.

Una carpeta genérica `Shared` tiende a convertirse en un contenedor sin
responsabilidad clara. Los elementos deberán ubicarse según su función:

- Regla o value object: Domain.
- Contrato o caso de uso: Application.
- Adaptador técnico: Infrastructure.
- Componente Unity o vista: Presentation.

Si aparece una necesidad transversal real, se evaluará un assembly específico
en el sprint correspondiente.

## Configuración relevante

### Auto Referenced desactivado

Los assemblies no son referenciados automáticamente por `Assembly-CSharp`.
Esto ayuda a detectar scripts colocados fuera de la estructura aprobada.

### No Engine References

Está activado en `Domain` y `Application`, por lo que el compilador impedirá el
uso accidental de UnityEngine o UnityEditor en esas capas.

### Referencias por nombre

S0.6 usa nombres de assembly para evitar distribuir archivos `.meta` generados
fuera del proyecto. Los nombres quedan congelados. Una migración futura a GUID
solo se realizará si ofrece una ventaja concreta.

## Fuera de alcance

- Código de gameplay.
- Composition root.
- Assembly de herramientas Editor.
- Referencias a paquetes concretos.
- InternalsVisibleTo.
- Analyzers y rulesets.
- Smoke tests; se implementan en S0.8.
